## ADR-002: API Versioning

### Context
The API will evolve over time and needs a versioning strategy.

### Decision
We will use **URL Path Versioning**: `/api/v1/`, `/api/v2/`

### Versioning Policy
- **Major version bump**: Breaking changes
- **Minor version bump**: New features (backward compatible)
- **Deprecation period**: 6 months minimum warning

### Consequences
- All API endpoints must include version prefix
- Multiple versions may run concurrently during transition

## ADR-003: Error Handling

### Context
Consistent error handling improves developer experience and debugging.

### Decision
We will use a standardized error response format:

```json
{
  "error": {
    "code": "ERR_{{DOMAIN}}_{{ERROR_TYPE}}",
    "message": "Human-readable message",
    "details": {},
    "timestamp": "ISO8601",
    "requestId": "uuid"
  }
}
```

### HTTP Status Code Usage
| Code | Usage |
|------|-------|
| 400 | Validation errors, bad request |
| 401 | Authentication required |
| 403 | Permission denied |
| 404 | Resource not found |
| 409 | Conflict (duplicate, version mismatch) |
| 422 | Unprocessable entity (business rule violation) |
| 500 | Unexpected server error |

## ADR-004: Naming Convention

### Context
Consistent naming improves code readability and maintainability.

### Decision

#### Files & Folders
- **Folders**: `kebab-case` (e.g., `user-management/`)
- **Files**: `kebab-case` (e.g., `user-service.ts`)
- **Test files**: `*.test.ts`, `*.spec.ts`

#### Code
- **Variables/Functions**: `camelCase` (e.g., `getUserById`)
- **Classes/Types**: `PascalCase` (e.g., `UserService`)
- **Constants**: `SCREAMING_SNAKE_CASE` (e.g., `MAX_RETRY_COUNT`)

#### API Endpoints
- **Resources**: `kebab-case` plural (e.g., `/api/v1/user-profiles`)
- **Actions**: Use HTTP verbs, not URL verbs
  - Good: `POST /users` (create)
  - Bad: `POST /create-user`

#### Database
- See: `database/standards.md`
